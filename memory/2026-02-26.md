# 2026-02-26 - Investigación Auto-Learning Service

## Problema Detectado

El Auto-Learning Service (`auto_learning_service.py`) se detiene repetidamente después de recibir señales SIGTERM/SIGINT.

### Patrón Observado (desde logs)
- 23:37:54 - Shutdown signal received
- 00:01:26 - Shutdown signal received (23 min después del restart)
- 00:02:39 - Shutdown signal received (1 min después del restart)

### Causa Raíz Identificada
El servicio `solana-jupiter-bot.service` systemd está ejecutando un watchdog simple que solo corre `master_orchestrator.py`. El `auto_learning_service.py` NO está siendo gestionado por este servicio.

El problema parece ser:
1. El `auto_learning_service.py` está siendo iniciado manualmente o por otro script
2. Algo está enviando señales SIGTERM al proceso
3. No hay integración formal entre el master orchestrator y el auto-learning service

### Estructura Actual
- **Systemd Service**: `solana-jupiter-bot.service` → `run_watchdog.sh` → `master_orchestrator.py`
- **Auto-Learning**: `auto_learning_service.py` (independiente, sin gestión formal)

### Archivos Clave
- `/etc/systemd/system/solana-jupiter-bot.service` - Servicio systemd
- `~/.openclaw/workspace/Solana-Cripto-Trader/run_watchdog.sh` - Watchdog simple
- `~/.openclaw/workspace/Solana-Cripto-Trader/master_orchestrator.py` - Trading principal
- `~/.openclaw/workspace/Solana-Cripto-Trader/auto_learning_service.py` - Auto-learning (problemático)

### Procesos Corriendo
- `master_orchestrator.py` (PID 710256) - OK
- `auto_learning_service.py` - NO corriendo (murió a las 00:02:49)

## Acciones Tomadas

1. Identificado el servicio systemd como posible causa de las señales
2. Detectado que el auto-learning service no está integrado con el sistema principal
3. Confirmado que solo el master_orchestrator está siendo gestionado formalmente

## Solución Propuesta

**OPCIÓN A (Recomendada)**: Integrar auto-learning DENTRO del master_orchestrator
- El auto-learning service debería ser un thread/subproceso del master orchestrator
- No necesita ser un proceso separado
- Simplifica la gestión y evita problemas de comunicación entre procesos

**OPCIÓN B**: Crear un servicio systemd separado para auto-learning
- Mantener como proceso independiente pero con gestión formal
- Requiere más configuración

**OPCIÓN C**: Usar el `start_auto_learning_trading.py` que integra ambos sistemas
- Este script ya existe y debería iniciar ambos sistemas correctamente
- Requiere actualizar el servicio systemd para usar este script en lugar del watchdog simple
